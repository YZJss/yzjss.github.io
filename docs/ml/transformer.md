# 第 23 章 大模型基础

这一章整理大模型相关的基础概念：GPT/BERT 区别、温度参数、MoE、后训练、强化学习、推理能力和幻觉。预训练模型如何迁移到任务放在第 22 章，Agent 系统放在第 24 章。

## 23.1 BERT 和 GPT 的区别

| 模型 | 结构 | 典型能力 |
| --- | --- | --- |
| BERT | Encoder-only | 文本理解、分类、抽取 |
| GPT | Decoder-only | 文本生成、对话、续写 |

BERT 更像“读懂一句话”，GPT 更像“根据上下文继续生成”。两者都来自 Transformer，但使用方式不同。

## 23.5 大模型的后训练

大语言模型通常先进行大规模预训练，然后经过后训练提高对话、指令遵循和推理能力。

常见阶段：

| 阶段 | 作用 |
| --- | --- |
| 预训练 | 学习通用语言和知识 |
| SFT | 学习按照指令回答 |
| 偏好对齐 | 让回答更符合人类偏好 |
| 强化学习 | 进一步优化目标行为 |

后训练不是重新学全部知识，而是在已有模型能力上调整输出行为和任务表现。

## 23.6 温度参数

生成模型常用温度控制输出随机性。

| 温度 | 输出特点 |
| --- | --- |
| 较低 | 更稳定、更保守 |
| 较高 | 更随机、更有创造性 |

温度越高，模型越可能选择低概率 token，因此创造性更强，但稳定性和可控性可能下降。

## 23.7 AI Agent

AI Agent 和普通聊天助手的区别在于是否具备闭环执行能力。

普通助手主要回答问题；Agent 通常能拆解任务、调用工具、执行操作、观察结果，并根据反馈继续行动。

一个简单 Agent 流程：

```text
接收任务 -> 规划步骤 -> 调用工具 -> 观察结果 -> 调整计划 -> 完成任务
```

Agent 的关键不只是语言理解，而是能把理解转化成可执行行动，并根据结果闭环修正。

## 23.8 使用预训练模型的注意事项

预训练模型不是万能的。

需要注意：

- 训练数据和目标任务是否相近。
- 是否存在数据泄漏。
- 微调数据是否足够干净。
- 模型输出是否稳定可靠。
- 是否需要领域知识和人工审核。

迁移学习的正确用法是利用已有表示，而不是盲目相信预训练模型一定适合所有任务。

## 23.9 预训练为什么有效

预训练有效的核心原因是：很多任务共享底层表示。

图像任务中，边缘、纹理、形状等特征在不同数据集之间可以复用。文本任务中，词义、语法、上下文关系也可以在不同任务之间复用。

预训练模型先在大规模数据上学到通用表示，再用少量任务数据调整到具体任务。

这比从零训练更稳，尤其适合标注数据少、模型较大的场景。

## 23.10 冻结哪些层

迁移学习中，一个常见问题是冻结多少层。

| 数据量 | 任务相似度 | 建议 |
| --- | --- | --- |
| 少 | 高 | 冻结大部分层，只训练分类头 |
| 少 | 低 | 冻结底层，微调高层 |
| 多 | 高 | 微调高层或全模型 |
| 多 | 低 | 可以全模型微调，但要小学习率 |

底层特征更通用，高层特征更任务相关。图像模型中，底层边缘纹理较通用；高层类别语义更依赖原始训练任务。

## 23.11 微调时的学习率

微调预训练模型时，学习率通常比从零训练更小。

原因是预训练参数已经有较好表示，过大的学习率可能破坏已有知识，这有时叫灾难性遗忘。

常见做法：

- 新增分类头使用较大学习率。
- 预训练主体使用较小学习率。
- 先冻结主体训练分类头，再逐步解冻。

## 23.12 BERT 微调流程

以文本分类为例，BERT 微调通常是：

```text
文本 -> tokenizer -> token ids -> BERT -> [CLS] 表示 -> 分类层 -> 类别概率
```

`[CLS]` 位置的输出常被当作整句表示，用于分类任务。

微调时需要注意：

| 项 | 说明 |
| --- | --- |
| max length | 太短会截断信息，太长会增加计算 |
| batch size | 受显存限制 |
| learning rate | 通常较小 |
| warmup | 训练初期稳定更新 |
| weight decay | 缓解过拟合 |

## 23.13 Embedding、Reranker 和生成模型

大模型应用里经常出现几类模型：

| 类型 | 作用 |
| --- | --- |
| Embedding 模型 | 把文本转成向量，用于召回和相似度计算 |
| Reranker 模型 | 对候选结果重新排序 |
| 生成模型 | 根据上下文生成答案 |

在 RAG 系统中，通常先用 embedding 从知识库召回候选文档，再用 reranker 精排，最后把文档交给生成模型回答。

Embedding 模型不是直接生成文本，它输出的是向量。Reranker 也不是负责长文本生成，而是判断 query 和文档的相关性。

## 23.14 Agent 的能力边界

Agent 的关键是闭环执行，但闭环不等于绝对可靠。

一个 Agent 至少需要：

- 理解目标。
- 拆解任务。
- 选择工具。
- 执行操作。
- 观察结果。
- 根据反馈修正。

风险也很明显：

- 工具调用可能失败。
- 计划可能分解错。
- 外部环境可能变化。
- 模型可能误判结果是否成功。

所以重要任务中，Agent 需要日志、权限控制、人工确认和回滚机制。

## 23.15 MoE 架构

MoE 是 Mixture of Experts，即混合专家模型。它不是让每个 token 都经过全部参数，而是通过路由机制选择一部分专家参与计算。

它的优势是：在相同推理成本下，模型可以拥有更大的总参数量，从而提升整体能力。

但 MoE 也会带来训练和部署难点，例如专家负载不均衡、通信开销、路由稳定性等。

## 23.16 强化学习和推理能力

大模型后训练中的强化学习不只是让模型“更会聊天”，也可能改变模型解决复杂问题的方式。

当奖励规则设计得足够清晰时，模型可以通过试错学会更长的推理链、更好的自我检查方式，以及更稳定的答案格式。

## 23.17 幻觉问题

幻觉指模型生成看起来合理但实际错误的内容。常见缓解方式包括：

- 检索增强，让模型回答前先查资料。
- 工具调用，让模型把计算、搜索、数据库查询交给外部工具。
- 引用来源，让回答可追溯。
- 对关键场景加入人工审核。

幻觉不能只靠一句提示词彻底解决，它是生成模型概率生成机制和训练数据限制共同导致的问题。
