# 第 24 章 AI Agent 与应用系统

AI Agent 是大模型落地应用中的重要方向。普通聊天助手主要回答问题，而 Agent 更强调“闭环执行”：能理解目标、拆解任务、调用工具、观察结果，并根据反馈继续行动。

## 24.1 Agent 和传统 AI 助手的区别

传统 AI 助手通常是：

```text
用户提问 -> 模型回答
```

Agent 更接近：

```text
用户目标 -> 任务规划 -> 工具调用 -> 结果观察 -> 调整计划 -> 完成任务
```

所以考试里问 Agent 和传统 AI 助手的核心区别，重点是是否具备闭环执行能力。

## 24.2 Agent 的基本组成

一个典型 Agent 系统通常包含：

| 模块 | 作用 |
| --- | --- |
| Planner | 把目标拆解成步骤，并选择后续行动 |
| Worker / Tool | 执行具体动作，例如搜索、写文件、查数据库 |
| Memory | 记录用户偏好、历史状态、长期信息 |
| Reflection | 检查结果是否满足目标，不满足则调整计划 |

这些模块不一定都由独立模型实现。它们更像是系统设计中的职责划分：规划负责决定路径，工具负责执行动作，记忆负责保存状态，反思负责检查结果。

## 24.3 Planner

Planner 是 Agent 的“大脑”。它负责把模糊目标拆成明确步骤。

例如用户说“帮我整理一份机器学习复习资料”，Planner 需要决定：

1. 先读取现有资料。
2. 提取考试题考点。
3. 对照参考书补缺漏。
4. 重构章节目录。
5. 写入 Markdown。
6. 构建和部署博客。

如果没有 Planner，模型只能被动回答，很难稳定完成多步骤任务。

## 24.4 Tool / Worker

Worker 提供具体能力，例如：

- 搜索网页。
- 读取文件。
- 执行命令。
- 操作浏览器。
- 调用数据库。
- 上传图片或部署网站。

工具调用让 Agent 从“只会说”变成“能做事”。但工具调用也带来风险，所以需要权限控制和操作确认。

## 24.5 Memory

Memory 用来保存上下文。短期记忆保存当前任务状态，长期记忆保存用户偏好、历史项目、常用配置。

没有 Memory 的 Agent 每次都像第一次见到用户，无法积累个人工作流。

## 24.6 OpenClaw 四角色

题库里提到的 OpenClaw 核心架构有四个角色：

| 角色 | 作用 |
| --- | --- |
| GateKeeper | 多渠道指令接收与身份验证 |
| Planner | 作为“大脑”负责任务拆解与决策规划 |
| Worker | 提供基础操作能力和具体任务方法 |
| Memory | 实现用户偏好和对话历史的持久化存储 |

记忆方式可以按一句话理解：GateKeeper 管入口，Planner 管计划，Worker 管执行，Memory 管长期状态。

## 24.7 RAG

RAG 是 Retrieval-Augmented Generation，检索增强生成。它的基本流程是：

```text
用户问题 -> 向量检索 -> 召回文档 -> 重排序 -> 拼接上下文 -> 生成回答
```

RAG 的目的不是让模型参数里记住所有知识，而是回答前先查外部资料。这样可以提高时效性，也能减少幻觉。

## 24.8 Embedding 和 Reranker 在 RAG 中的作用

Embedding 模型负责把问题和文档编码成向量，用于快速召回相似内容。

Reranker 负责对召回结果重新排序，判断哪些文档和问题最相关。

常见流程：先用 embedding 粗召回，再用 reranker 精排，最后交给生成模型回答。

## 24.9 Agent 的落地场景

Agent 最容易先落地在流程化、规则密集、结果可验证的场景，例如：

- 内部知识库问答。
- 客服工单处理。
- 数据报表生成。
- 代码辅助和自动化测试。
- 运营流程自动化。

越开放、越高风险、越难验证的任务，越需要人工审核和权限约束。

## 24.10 Agent 的风险

Agent 的风险主要来自“会执行”。常见问题包括：

- 计划错误。
- 工具调用错误。
- 权限越界。
- 长期记忆污染。
- 执行成本不可控。
- 错误结果被继续当成事实使用。

因此工程上通常需要日志、权限隔离、人工确认、失败回滚和敏感操作保护。
